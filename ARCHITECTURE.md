## Architectural Design Decisions & "The Why"

### 1. The Shared `DecodedInstruction` Struct (IR)
Instead of having the Assembler pass messy strings around, and instead of having the CPU decode bits directly in its `execute()` methods, we created a single `DecodedInstruction` struct in the `ISA/` folder. 
* **For the Assembler:** It acts as an Intermediate Representation (IR). The Assembler parses text into this struct, and passes the struct to the `InstructionPacker`.
* **For the CPU:** The `Decoder` takes raw binary, unpacks it into this struct, and passes the struct to the `CPU`.
* **Why:** This enforces a strict separation of concerns and prevents circular dependencies between the software toolchain and the hardware simulator.

### 2. The `Bus` Interconnect
Rather than giving the CPU a direct pointer to `Memory`, we built a `Bus` class. 
* **Why:** In real silicon, the CPU does not talk directly to RAM chips; it talks to a memory controller via a bus system. This decoupled architecture allows us to easily add Memory-Mapped I/O (like keyboards or displays) in the future without modifying the CPU.

### 3. Separation of `Assembler` and `InstructionPacker`
* **Why:** Assembly parsing involves messy string manipulation, tokenization, and label tracking. Machine code generation involves intense bitwise shifting (`<<` and `|`) and masking. Keeping these in separate files ensures the codebase remains readable. The Assembler handles "Software", while the Packer acts as the "Hardware Encoder".

### 4. The Two-Pass Assembler
* **Why:** RISC-V assembly frequently jumps *forward* to labels that haven't been defined yet (`BEQ x1, x2, FUTURE_LABEL`). A single-pass compiler would crash here. The First Pass strictly calculates memory addresses and records label locations. The Second Pass does the actual instruction translation using the pre-built Symbol Table.

### 5. `InstructionType::I_SHIFT`
* **Why:** `SRAI`, `SRLI`, and `SLLI` are technically I-Type instructions, but they uniquely split their 12-bit immediate into a 5-bit shift amount (`shamt`) and a 7-bit identifier (`funct7`). Instead of writing "Jugaad" (hacky) `if` statements in the parser, we added `I_SHIFT` as a dedicated enum type. This elegantly forces the parser and packer to handle the split fields correctly without polluting the standard `I-Type` logic.

### 6. The Automated Testbench (`main.cpp`)
* **Why:** Hardware verification is rigorous. The `main.cpp` loop reads 15 distinct `.asm` files from `programs.txt`. For every file, it instantiates a **brand new** Motherboard (`Memory`, `Bus`, `CPU`), compiles the code, loads it, and runs it until `ECALL` halts the CPU. It then checks if `x31 == 1` (Pass) or `0` (Fail). This guarantees complete state isolation between tests and immediately catches regressions.

## Pipeline Flow (Software to Hardware)
1. **Developer:** Writes `test.asm`.
2. **Assembler:** Reads `test.asm`, parses text, resolves labels, and generates an 8-character hex string per instruction.
3. **InstructionPacker:** Shifts the parsed fields (`opcode`, `rs1`, `imm`) into standard RISC-V binary formats.
4. **Assembler:** Saves the binary formats to `test.hex`.
5. **Loader:** Reads `test.hex` and writes the bytes into `Memory` via the `Bus`.
6. **CPU (`step`):** Fetches the 32-bit word from the `Bus`.
7. **Decoder:** Slices the 32-bit word back into a `DecodedInstruction` struct.
8. **CPU/ALU:** Executes the instruction, updates Registers, and advances the PC.
